# /qtim:team-lazy — лёгкий режим без warm-up

> Читает только team-lead. Канон рантайма (`Agent`/`SendMessage`/`Task*`, упразднённые `TeamCreate`/`team_name`) и Decision Matrix — в [`team-up.md`](team-up.md). Здесь — отличия lazy-режима.

## Когда

Режим **C** из Decision Matrix: несколько слоёв за один проход БЕЗ петель доводки impl↔test↔review. Средняя фича (например миграция + UI), баг-фикс на 2-3 роли, fan-out-аудит. Платишь только за реально поднятых членов.

## Как

1. **Команду НЕ прогреваешь.** Она существует неявно (сессия = одна команда). Ничего не создаёшь.
2. **Спавнишь роль в момент необходимости** через `Agent({ name, subagent_type, prompt })` — `subagent_type` и `read on spawn` бери из `.claude/team-charter.md`. Прочитай charter, если он ещё не в контексте. Если задача ссылается на фичу из `docs/features/<slug>/` — её `plan.md`/`prd.md` являются источником scope; по завершении обнови Status артефактов.
3. **Каждый член** при старте читает CLAUDE.md + свою секцию charter + свои `read on spawn`, отрабатывает задачу.
4. **Следующий шаг цепочки** — `SendMessage({ to: "<name>" })` (контекст члена сохранён), а не повторный `Agent` (тот стартует с нуля).
5. **Параллельный fan-out** — несколько `Agent` одним сообщением, если единицы работы независимы.

## Критерий переключения C → D

Как только появляется **петля обратной связи** (tester нашёл баг → исполнитель правит → re-test → reviewer вернул на доработку), ты фактически в режиме **D**. Не отменяй уже поднятых — просто доспавни недостающих обычными `Agent`-вызовами (team одна на сессию, дублей не будет) и веди общий `Task*` list. Эскалация вверх дешёвая — это и есть причина начинать с lazy при сомнении.

## Anti-patterns

- Спавнить всех «на всякий случай» — это уже `/qtim:team-up`, а не lazy.
- Повторный `Agent` с тем же `name` для продолжения — теряешь контекст; используй `SendMessage`.
- Тащить orchestration-логику в промпты членов — она только у team-lead.
