# /qtim:team-up — собрать полную команду

> **Этот файл читает только team-lead (оркестратор).** Субагенты его не открывают — orchestration-логика не должна попадать в их prompt'ы во избежание рекурсии.
>
> Команда и её состав описаны в `.claude/team-charter.md` проекта (его генерирует `/qtim:setup`). Если charter отсутствует — сначала запусти `/qtim:setup`.

---

## Модель оркестрации (рантайм Agent Teams)

Актуальный рантайм Claude Code. Держи в голове перед любым режимом:

- **Сессия = одна неявная команда.** Примитив `TeamCreate` упразднён — регистрировать/создавать команду не нужно, она существует имплицитно.
- **Член команды** = субагент через `Agent({ name, subagent_type, prompt })`; `name` делает его адресуемым.
- **`team_name` в `Agent`** — deprecated, рантаймом игнорируется. Не передаём.
- **Продолжение** поднятого агента (контекст сохранён) — `SendMessage({ to: "<name>", … })`. Повторный `Agent` с тем же именем = старт с нуля; не плодим.
- **Фоновые роли** — `Agent({ …, run_in_background: true })`: работают асинхронно, уведомление по завершении.
- **Общий task list** — `Task*` (`TaskCreate`/`TaskUpdate`/`TaskList`/…), сессионный, без привязки к team_name.
- **«idle»** в старом смысле (член висит в процессе) упразднён. Эквивалент: агент завершил warm-up/задачу и остаётся **продолжаемым через `SendMessage`**. «Иди в idle, жди задач» читается как «заверши warm-up коротким ack».
- **Liveness** = спавнил ли ты агента в **текущей** сессии (не файл на диске). In-process агенты НЕ переживают рестарт CLI.
- **Shutdown** — `TeamDelete` упразднён; прервать фоновых = `TaskStop`. Стирать с диска нечего.

## Decision Matrix — какой режим выбрать ДО запуска

Когда приходит задача, **сначала выбери режим, потом действуй.** Прогрев членов дорогой — слепо поднимать полную команду нельзя. Принцип: **start cheap, эскалируй вверх по факту.** Границы между уровнями намеренно нестрогие — эскалация в середине дешёвая, поэтому при сомнении бери уровень НИЖЕ.

### Одна ось — «глубина координации»

Уровень определяется НЕ числом файлов и НЕ числом ролей, а **глубиной координации**: сколько обособленных единиц работы и есть ли петли обратной связи (impl↔test↔review).

| Режим | Глубина координации | Сигнал в задаче |
|---|---|---|
| **A. Direct** | агентов нет | Team-lead закрывает сам за несколько инструментов: понятная правка, вопрос, поиск в известном месте. Контекст не засоряется, параллелить нечего. |
| **B. Single agent ad-hoc** | 1 агент, fire-and-forget, без обратной связи | Одна обособленная единица работы за один проход. **Или** нужна изоляция контекста — даже если работа мелкая. |
| **C. `/qtim:team-lazy` + dynamic spawn** | N агентов БЕЗ review-fix циклов: линейный pipeline или параллельный fan-out | Несколько слоёв за один проход; результат НЕ гоняется по кругу impl↔test↔review. Спавнишь по требованию — платишь только за поднятых. |
| **D. Полный `/qtim:team-up`** | итеративные петли impl↔tester↔reviewer | Есть цикл «сделал → проверили → вернули на доработку → перепроверили»: полный эпик design→impl→test→review. Прогрев всех членов окупается только здесь. |

**Граница C↔D — это наличие петли обратной связи, а не число ролей.** 3 роли линейно (один проход) = **C**. Те же 3 роли с возвратами (tester нашёл баг → исполнитель правит → re-test) = **D**.

### Ортогональные драйверы (поднимают уровень независимо от объёма)

- **Изоляция контекста** → минимум **B**, даже если работа «мелкая». Широкий поиск / research / разбор многих файлов, чей выхлоп засорил бы главный контекст, отдавай субагенту — он вернёт ВЫВОД, не сырьё.
- **Параллелизм** независимых единиц → **C** (fan-out) или Workflow-паттерн (см. ниже).
- **Риск / необратимость / задет инвариант** (авторизация, деньги, публичный контракт, миграции данных) → режим A/B/C/D НЕ меняет, но требует **дизайн-фазу + approval-гейт**. За эту ось отвечает [`intake-protocol.md`](../reference/intake-protocol.md) (§ тест «развилка?»).

### Режимы детально

**A. Direct (без агентов).** Самые тривиальные случаи. Team-lead сам через `Read`/`Edit`/`Bash`. Примеры: «где определена функция X», «исправь опечатку», «прочитай файл и расскажи что внутри».

**B. Single agent ad-hoc.** Один `Agent` через `subagent_type` под задачу:
- `Explore` — поиск по репо без правок (классический случай изоляции контекста).
- `general-purpose` — задача в несколько слоёв, но в одну итерацию без обратной связи.
Спавн с явной задачей → отработал → завершился. Ни ack, ни coordination.

**C. `/qtim:team-lazy` + dynamic spawn.** Команда существует неявно — ничего не создаёшь, спавнишь только нужных через `Agent`. Каждый читает свои `read on spawn` из charter, отрабатывает; следующий цикл — `SendMessage`. Линейно или параллельно, но БЕЗ петель доводки.

**D. Полный `/qtim:team-up`.** Все члены сразу в warm-up. Максимальный старт-overhead, окупается на больших эпиках с циклами доводки.

### Движение между уровнями

- **Эскалация вверх (главный приём).** Выбрал **B**, понял что нужна тестировка → доспавни `tester` отдельным `Agent`. **C** разрослась в эпик с циклами → доспавни остальных обычными `Agent`-вызовами (team одна на сессию, дублей не будет). **Уже поднятых НЕ перезапускай** — продолжай через `SendMessage`.
- **Деэскалация.** Подняли с запасом, хватило меньшего — лишних не спавни; ставших ненужными фоновых гаси `TaskStop`.
- **Не-агентный выход (abort).** Если задача требует РЕШЕНИЯ пользователя, а не исполнения — это стоп: вынести в brief / вопрос по [`intake-protocol.md`](../reference/intake-protocol.md), агентов не поднимать.

### Anti-patterns

- **Full команда ради одного `Explore`** — это **B**.
- **Direct, когда впереди петли доводки разных слоёв** — это **D**, а не A.
- **Speculative warm-up** — поднимать всех «потому что потом будет много». Поднимай под ТЕКУЩИЙ объём; следующий эпик доспавнит себя сам.
- **Выбор режима числом ролей.** Меряй глубиной координации (петли), не «сколько ролей».
- **Путать `/qtim:team-lazy` с `/qtim:team-up`** — `lazy` без прогрева, члены по ходу; `up` — прогрев всех сразу.

### Второе измерение — форма оркестрации (паттерны)

A/B/C/D отвечают на «**сколько** оркестрации». Ортогонально — «**какая форма**»: некоторым задачам нужен детерминированный control flow (выбор лучшего из N, цикл до сходимости, широкий fan-out, состязание исполнитель↔оппонент). Каталог из 6 паттернов — [`orchestration-patterns.md`](../reference/orchestration-patterns.md).

| Сигнал в задаче | Паттерн |
|---|---|
| «несколько подходов, выбрать лучший» | Tournament |
| «flaky / падает раз в N / крутить пока не сойдётся» | Loop until done |
| «разобрать пачку входящих, маршрутизировать» | Classify-and-act |
| «проверить N таблиц/роутов/файлов параллельно → один вывод» | Fan-out-and-synthesize |
| «security-critical / надо сломать вывод» | Adversarial (claude+codex) |
| «сгенерить много вариантов, отобрать топ-K» | Generate-and-filter |

Большинство задач паттерна **не** требуют. Паттерны 1/2/4/5/6 исполняются движком **Workflow** (требует opt-in пользователя).
**Порядок выбора:** (1) глубина координации A/B/C/D → (2) ортогональные драйверы → (3) нужна ли форма-паттерн.

## Если выбран режим D — выполни шаги по порядку

> **Universal baseline для ВСЕХ членов** (включая team-lead). При spawn'е каждого добавляй в начало prompt'а:
> «**Universal skills (mandatory)**: `using-superpowers` (invoke перед первым ответом для активации остальных skills) + `karpathy-guidelines` (поведенческие гайдлайны при любом коде/ревью). Эти ДО специализированных skills.»

## 1. Прочитай charter

Прочитай `.claude/team-charter.md` целиком — это контракт команды: состав ролей, их `subagent_type`, `read on spawn`, зоны ответственности, доменные инварианты. Если содержимое уже в контексте — не перечитывай.

## 2. Проверь liveness

Liveness определяется **фактом спавна в текущей сессии**, а не файлом на диске.
- Если ты сам спавнил членов **в текущей сессии** — команда жива: сообщи «уже инициализирована» и не перезапускай (продолжай через `SendMessage`).
- Если **в этой сессии** не спавнил (новый запуск CLI) — членов нет: спавни заново. Не слать `SendMessage` «членам» из прошлой сессии — их контекст мёртв.

## 3. Команда существует неявно — создавать нечего

`TeamCreate` упразднён: сразу к спавну (шаг 4).

## 4. Спавни членов параллельно

Отправь **одним сообщением** N вызовов `Agent` (concurrency), каждый с `run_in_background: true`. Каждый получает `name`, `subagent_type` и компактный system prompt. **Состав, типы и `read on spawn` бери из charter проекта** — не хардкодь здесь.

Канонический (дефолтный) состав — **фактический бери из charter проекта**, он может быть короче (Compact-проект). Имена ролей зафиксированы, задачи привязаны к `owner`:

**Core (per-эпик flow):** `architect` · `db` · `front` · `tester` · `reviewer` · `explorer`
**Cross-cutting (async):** `devops` · `product` · `auditor`

Шаблон промпта члена (компактный — детали роль читает сама из charter + memory, не дублируй charter в промпт):

```
Agent({
  name: "<role>",
  subagent_type: "<из charter>",
  run_in_background: true,
  prompt: "Ты <роль> команды <project>.

  Universal skills (mandatory): using-superpowers (invoke перед первым ответом) + karpathy-guidelines. ДО специализированных.

  ПРИ СТАРТЕ прочитай в порядке:
  1. CLAUDE.md
  2. .claude/team-charter.md — свою секцию '<role>' + 'Инварианты домена'
  3. свои файлы из 'read on spawn' этой секции charter (memory/* и т.п.)

  Работай строго в своей mission (см. do-not-touch в charter). Между членами — SendMessage; задачи — TaskCreate/TaskUpdate(owner). Codex second-opinion на gate-точках — по reference/codex-consult.md (advisory; инвариант > codex; read-only; fail-soft), протокол читай на gate-точке.

  После прочтения — короткий ack '<role> ready' и оставайся продолжаемым через SendMessage."
})
```

> Роли с особыми `subagent_type` (`explorer` → `Explore`; `devops`/`product`/`auditor` → `general-purpose` если генератор не создал им выделенный тип) — как прописано в charter.

## 5. Подтверди готовность

После того как все члены завершили warm-up и прислали ack:

```
TaskList()   ← покажи текущий список задач (если есть)
```

Ответь пользователю: список поднятых членов (role — subagent_type — ready), сводку `TASKS: N pending / M in_progress / K completed`, «Готов принимать задачи».

## Важно

- Если член не стартанул — не падай, продолжай с остальными, сообщи пользователю.
- Спавн всех делается **одним сообщением** с параллельными `Agent` tool calls.
- Промпты членов выше — минимум. Не дублируй charter в каждый промпт — пусть читают сами.
