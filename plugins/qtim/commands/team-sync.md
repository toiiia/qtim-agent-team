# /qtim:team-sync — миграция команды на текущую версию плагина

> Читает только team-lead. Запускается в проекте с уже собранной командой, когда плагин
> обновился (SessionStart-hook подсказывает это при дрейфе версий). Не путать с
> [`setup.md`](setup.md) — тот собирает команду с нуля и задаёт вопросы; sync только подтягивает
> изменения движка в уже сгенерированное. Нет charter — сначала `/qtim:setup`.

## Контракт: что менять, что сохранять

- **Engine-managed (обновляется из плагина):** `tools:` frontmatter агентов, канонические блоки
  шаблонов (header-note, memory-блок, codex-указатели), состав секций charter, путь к протоколу
  codex-consult, `permissions.deny`-baseline, дубли hooks.
- **Проектная конкретика (сохраняется как есть):** подставленные значения плейсхолдеров (стек,
  команды build/test), доменные инварианты, состав и имена ролей, подобранные skills,
  intake-режим, переопределённые `model`, весь `memory/` и `.claude/agent-memory/`.
- Ничего не удалять молча. Конфликт (пользователь руками правил engine-часть) — показать
  и спросить, не перезаписывать.

## Шаги

1. **Версии.** Прочитай `version` из `${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` и штамп
   `generated-by: qtim vX.Y.Z · mode: …` из `.claude/team-charter.md`. Плагин не установлен /
   его каталог недоступен → сообщи, что sync требует установленного плагина (подтягивать нечего),
   и остановись. Штампа нет → считай версию charter `< 1.3.0`; режим определи по факту
   (`.claude/commands/team-up.md` существует → standalone, иначе plugin-linked). Версии
   совпадают → сообщи «уже синхронизировано» и остановись (продолжай только по явной просьбе
   пользователя форсировать). Версия charter новее плагина (downgrade) → диапазон миграций пуст:
   обнови только штамп и явно сообщи об откате.
2. **Реестр миграций.** Прочитай [`../reference/migrations.md`](../reference/migrations.md)
   и возьми все записи от версии charter (исключительно) до текущей версии плагина
   (включительно), по порядку.
3. **Diff-анализ.** Для каждой затронутой записями сущности сравни текущее состояние проекта
   с актуальным каноном: `.claude/agents/*` ↔ `${CLAUDE_PLUGIN_ROOT}/agents/*` (сравнивай каркас
   и engine-блоки, а не подставленные значения), charter ↔ структура секций из setup Phase 4.1,
   `.claude/settings.local.json` ↔ baseline из setup Phase 4.3.
4. **План + подтверждение.** Покажи компактный план: файл → что изменится → из какой записи
   миграции. **Ничего не пиши до подтверждения пользователя.**
5. **Применение.** Внеси изменения. При standalone дополнительно пере-скопируй движок
   (`.claude/commands/`, `.claude/reference/`, `.claude/workflows/`) из плагина, поправив
   ссылки и имена команд (правила — setup Phase 4.6).
6. **Штамп.** Обнови в шапке charter `generated-by:` на текущую версию плагина (mode сохрани;
   если штампа не было — добавь).
7. **Отчёт.** Что мигрировано, что пропущено и почему (конфликт / решение пользователя),
   что осталось сделать руками.

## Critical Rules

1. **`memory/` и `.claude/agent-memory/` — данные проекта:** не менять, не «чистить», не
   реорганизовывать. Sync мигрирует механику, не знания.
2. **Не задавай заново вопросы setup (Q1-Q8)** — ответы уже материализованы в charter. Если
   миграция вводит новую секцию, которой нет (например «Intake-режим» в старом charter), —
   один точечный вопрос или дефолт с пометкой в плане.
3. **Канон рантайма** действует и здесь: никаких `TeamCreate`/`TeamDelete`/`team_name`
   в мигрированных файлах.
4. **Плейсхолдеры `{{...}}` в проект попадать не должны:** из шаблонов переноси каркас
   и engine-блоки, значения бери из текущих файлов проекта.
5. **Только чтение и запись markdown/json** — никаких lint/tests/migrations/git в процессе sync.
