# Intake-протокол оркестратора — «автопилот»

> Generic reference плагина agent-team. Проектные инварианты и пути живут в charter проекта
> (генерируется setup); здесь — переносимая механика.

> Как team-lead принимает задачу пользователя и когда эскалирует к нему. Читает **только
> оркестратор**. `Classify` синхронизирован с decision matrix (`../commands/team-up.md`): режим
> A/B/C/D определяется ТАМ по «глубине координации» + драйверам изоляции/параллелизма; intake
> владеет ортогональной осью риск/обратимость → дизайн-фаза, режим не дублирует. Источник
> предпочтения — charter проекта (как дефолты по стилю участия). Связан с
> `../commands/team-up.md` (decision matrix), `orchestration-patterns.md` (формы), charter
> проекта (роли/инварианты).

## Принцип

**Участие пользователя асимметрично по фазам.**

- **Анализ + проектирование — пользователь в контуре активно.** Оркестратор НЕ кодит сразу;
  выносит разбор задачи и варианты, пользователь дополняет доменным контекстом и требованиями,
  которые оркестратор знать не может. Здесь участие пользователя — ценность, а не издержка.
  Это диалог с итерациями, а не один точечный вопрос.
- **Реализация (код / тесты / ревью) — автопилот, минимум пользователя.** После approval —
  тишина, прерывание только на вновь всплывшем необратимом.
- **Граница — явный approval-гейт** по согласованному дизайну.

Пользователь формирует **ЧТО** и общий подход; оркестратор сам выбирает режим (A/B/C/D),
агентов и форму (штатный флоу / паттерн), реализует **КАК** и доводит до результата. Мелочи
реализации закрываются дефолтом, не выносятся.

## Что передаёт пользователь

Минимум — **цель своими словами**. Без режима/агентов/паттернов (их выбирает оркестратор).
Опциональные рычаги-модификаторы (одно слово в промпте):

| Слово | Эффект |
|---|---|
| `ultracode` | разрешена масштабная оркестрация (Workflow, десятки агентов) |
| «только спроектируй» / «не кодь» | стоп после ADR — принести план |
| «MVP» / «по-быстрому» | минимальный режим, меньше агентов |
| «это прод» / «осторожно» | поднять порог эскалации — спрашивать чуть больше |
| «не спрашивай, реши сам» | закрывать даже развилки дефолтом (на риск пользователя) |

## Пайплайн оркестратора (на каждую задачу)

1. **Classify.** Тип (фича / баг / рефактор / аудит / дизайн), риск, **обратимость**.
   → режим A/B/C/D + форма-паттерн берутся из [`../commands/team-up.md`](../commands/team-up.md)
   (ось «глубина координации» + драйверы изоляции/параллелизма) — здесь НЕ дублируем. Твоя ось
   на этом шаге — **риск/обратимость → нужна ли полноценная дизайн-фаза** (глубина ∝ сложности:
   баг / опечатка / мелкая правка → сразу в Execute, дополнять нечего; фича / изменение схемы /
   новая поверхность → дизайн-фаза обязательна).
2. **Design — совместная фаза, пользователь в контуре.** Для нетривиального НЕ кодить сразу.
   `architect` + `brainstorming` (mandatory: вытащить open questions + user intent), при
   необходимости tournament / `grill-me` как вход. Принести пользователю **design brief**:
   понимание задачи, объём / слои / риски, затронутые доменные инварианты проекта (определены в
   charter + memory/), 2-3 варианта подхода с трейд-оффами, предположения и открытые вопросы.
   Пользователь дополняет и правит — допускаются итерации. Это диалог, а не один batch-вопрос;
   на проактивность здесь не экономить.
3. **Approval-гейт.** Согласованный план/ADR зафиксировать в `memory/` (decisions). Дальше —
   реализация без возврата к пользователю (кроме вновь всплывшего необратимого). Механизм для
   нетривиальных задач — **plan mode** (`EnterPlanMode` → план → approval пользователя →
   `ExitPlanMode`).
4. **Execute — автопилот.** Штатный Team-флоу или Workflow-паттерн. До конца: impl → tester →
   (UI? real-browser QA-gate) → reviewer → коммит синтеза/решений в `memory/`. **Без вопросов**
   (кроме одного точечного чек-поинта, если необратимое всплыло позже, чего не было видно на
   дизайне).
5. **Report.** Результатом, не процессом. Что сделано, что зафиксировано в `memory/`, остатки.

## Тест «развилка?» — что обязательно проговорить с пользователем

Двойное назначение: (а) что **обязательно** вынести в design brief на фазе 2 (даже если кажется,
что есть разумный дефолт); (б) что **прерывает автопилот** на фазе 4, если всплыло уже после
approval. Триггер — выполнено **хотя бы одно**:

- **Необратимо / дорого откатить:** миграция с трансформацией данных, удаление, смена
  публичного API-контракта, деньги/биллинг, ротация секретов.
- **Задет доменный инвариант проекта** (определён в charter + memory/) или модель видимости /
  авторизации — это продуктовое правило, не имплементационная деталь.
- **Неоднозначные требования:** задача допускает ≥2 прочтения → разные продукты. Гадать нельзя.
- **Продуктовая развилка:** варианты не эквивалентны и влияют на то, что видит/делает
  пользователь (не просто стиль кода).
- **Новый класс поверхности/актора** — security-граница (новый публичный endpoint, новая роль,
  новый канал доступа к данным).
- **Конфликт с зафиксированным решением** в `memory/` — подсветить, не «исправлять молча».

**НЕ развилка (решаю сам, без вопроса):**

- Имена, структура файлов, выбор агента, выбор паттерна, порядок шагов.
- Эквивалентные тех-решения без продуктового эффекта.
- UI-копи и тексты — генерю + фильтрую по рубрике, показываю в результате (generate-and-filter).
- Стиль, форматирование, какой «пилот» запустить, как отчитаться.
- Обратимые правки в рамках уже принятого ADR.

## Правила вопросов (по фазам)

- **Дизайн-фаза — проактивный диалог.** Выносить разбор и варианты пользователю, давать
  дополнить. Здесь его вклад ценен — на проактивность НЕ экономить. Структурированный выбор —
  через `AskUserQuestion` с рекомендованным дефолтом первым (если пользователь склонен принимать
  дефолты — см. charter/feedback), но открытое обсуждение brief'а уместнее жёсткого batch'а.
- **Фаза реализации — тишина.** После approval не возвращаться к пользователю; исключение —
  один точечный вопрос на вновь всплывшем необратимом.
- **Глубина ∝ сложности.** Тривиальное (баг / опечатка / мелкая правка) идёт в Execute сразу,
  без дизайн-фазы — там дополнять нечего.

## Anti-patterns

- ❌ Начинать кодить нетривиальную задачу без design brief и approval пользователя (он добавляет
  ценность на дизайне — не обходить эту фазу).
- ❌ Устраивать дизайн-фазу/brief на тривиальной правке (баг / опечатка) — идти сразу в Execute.
- ❌ Капать мелкими вопросами **в фазе реализации** (после approval).
- ❌ Спрашивать на реализации то, что закрывается дефолтом из `memory/` / конвенций проекта.
- ❌ Выносить на пользователя выбор режима/агента/паттерна — это работа оркестратора.
- ❌ Гадать на необратимом/неоднозначном вместо того, чтобы вынести это в brief на дизайне.
- ❌ Отчитываться процессом («поднял agent X, он сделал Y…») вместо результата.
- ❌ Молча исправлять конфликт с зафиксированным в `memory/` решением.
