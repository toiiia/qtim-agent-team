# Debug-loop — дисциплина диагностики

> Скил плагина qtim; доступен любой роли через `Skill`. Адаптация `diagnosing-bugs`
> из [mattpocock/skills](https://github.com/mattpocock/skills) (MIT).

Для нетривиальных багов: причина не видна сразу, баг плавающий, или «вчера работало».
Тривиальное (опечатка, очевидный null) чини сразу — эта дисциплина не для него. Фазы
пропускай только с явным обоснованием. Перед стартом прочитай контекст проекта
(`memory/`: инварианты, карта архитектуры) — модель слоёв сужает пространство гипотез.

## Фаза 1 — построй feedback loop (это и есть скил)

Всё остальное — механика. Есть тесный сигнал pass/fail, краснеющий на ЭТОМ баге, — причина
найдётся: бисекция, гипотезы и инструментирование просто потребляют его. Нет сигнала —
никакое чтение кода не спасёт. Трать здесь непропорционально много усилий.

Способы построить (в порядке предпочтения):

1. Падающий тест на том шве, до которого дотягивается баг (unit / integration / e2e).
2. curl/HTTP-скрипт против запущенного dev-сервера.
3. CLI-вызов с фикстурой, diff stdout против эталона.
4. Сценарий в headless-браузере: assertions на DOM / console / network.
5. Replay захваченного артефакта: сохранённый payload / лог событий прогоняется через
   изолированный код-путь.
6. Одноразовый harness: минимальный кусок системы (один сервис, замоканные зависимости)
   вызывает баг-путь одной функцией.
7. Бисекция: баг появился между двумя известными состояниями (коммит, датасет, версия) —
   автоматизируй «встань на X, проверь» под `git bisect run`.

Получив рабочий loop — ужимай его: быстрее (кэшируй setup, сузь scope), острее (assert на
конкретный симптом, не «не упало»), детерминированнее (зафиксируй время, seed RNG, изолируй
файловую систему и сеть). Двухсекундный детерминированный loop — суперсила; тридцатисекундный
флаки — почти бесполезен.

**Недетерминированный баг:** цель — не чистое репро, а частота воспроизведения, с которой
можно работать. Гоняй триггер циклом 100×, параллель, добавляй стресс, сужай тайминги,
вставляй sleep'ы. 50 % воспроизводимости дебажится; 1 % — нет. Поднимай частоту, пока не
станет дебажимо.

**Loop не строится** — остановись и скажи явно: что перепробовал и что нужно (доступ к
воспроизводящему окружению, захваченный артефакт — HAR / лог / дамп, разрешение на временное
инструментирование). Субагент эскалирует team-lead'у через `SendMessage`, team-lead —
пользователю. К гипотезам без loop не переходить.

**Критерий завершения фазы:** одна команда (путь скрипта / вызов теста / curl), уже
прогнанная хотя бы раз, которая (а) ловит именно симптом пользователя — красная на этом
баге, зелёная после фикса, а не «отработала без ошибок»; (б) даёт один и тот же вердикт на
каждый прогон; (в) отрабатывает за секунды; (г) запускается без человека. Поймал себя на
«читаю код и строю теорию», а команды нет — стоп: это ровно тот провал, от которого скил.

## Фаза 2 — воспроизведи и минимизируй

Прогони loop, посмотри на красное. Убедись, что это симптом, о котором говорил пользователь,
а не соседний баг: не тот баг = не тот фикс. Затем ужимай репро до минимального красного:
убирай входы, вызовы, конфиг, данные и шаги по одному, перепрогоняя loop после каждого
выреза. Готово, когда каждый оставшийся элемент несущий: убери любой — loop зеленеет.
Минимальное репро сужает пространство гипотез и становится регрессионным тестом фазы 5.

## Фаза 3 — гипотезы

3–5 ранжированных гипотез ДО проверки первой: одиночная гипотеза якорит на первой
правдоподобной идее. Каждая фальсифицируема — формулирует предсказание: «если причина X,
то изменение Y уберёт баг / изменение Z усилит его». Не можешь сформулировать предсказание —
это вайб, не гипотеза: выброси или заостри. Ранжированный список покажи (субагент —
team-lead'у, team-lead — пользователю): у них бывает знание, мгновенно перестраивающее
ранжирование («мы вчера деплоили ровно это»). Не блокируйся на ответе — нет реакции,
работай по своему ранжированию.

## Фаза 4 — инструментируй

Каждый probe отвечает конкретному предсказанию из фазы 3. Одна переменная за раз.
Предпочтение инструментов: debugger / REPL, если окружение позволяет (один breakpoint лучше
десяти логов) → точечные логи на границах, различающих гипотезы → никогда «залогируй всё
и грепай». Каждый debug-лог — с уникальным тегом (`[DEBUG-a4f2]`): cleanup потом — один
grep; логи без тега переживают отладку и остаются мусором.

**Перф-ветка:** для регрессий производительности логи обычно врут. Сначала baseline-замер
(timing harness, профайлер, план запроса), потом бисекция. Мерить до фикса, не после.

## Фаза 5 — фикс + регрессионный тест

Регрессионный тест — ДО фикса, но только на правильном шве: тест воспроизводит паттерн бага
как он случается на реальном call site. Если правильного шва нет (баг требует цепочки
вызовов, которую unit-тест не повторит) — это само по себе находка: архитектура мешает
запереть баг; зафиксируй в отчёте / задаче на `architect` (режим REVIEW), не пиши тест ради
галочки. Со швом: минимальное репро → падающий тест → смотри, как падает → фикс → смотри,
как проходит → перепрогони loop фазы 1 на исходном (не минимизированном) сценарии.

## Фаза 6 — cleanup

- [ ] Исходный сценарий не воспроизводится — loop фазы 1 зелёный
- [ ] Регрессионный тест в сьюте (или отсутствие правильного шва зафиксировано)
- [ ] Всё инструментирование снято — grep по `[DEBUG-` пуст
- [ ] Одноразовые harness'ы удалены
- [ ] Подтвердившаяся гипотеза названа в коммите/отчёте — следующий отладчик учится на ней

Затем спроси: что предотвратило бы этот баг? Если ответ архитектурный (нет шва для теста,
скрытая связность, спутанные вызовы) — задача `architect`'у в режим REVIEW. После фикса,
не до: сейчас информации больше, чем на старте.
